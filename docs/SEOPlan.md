## 2. Guiding principles for every change

1. **No rebuild, no template regression.** We extend the current Next.js App Router structure, component library, and design tokens (`brand-primary` blue, `brand-accent` orange, existing card/section patterns, reveal animations). Nothing gets rebuilt from scratch.
2. **Every service page is hand-written, not templated.** Shared *layout* (hero, sections, FAQ, CTA) is fine and expected for consistency — shared *copy* is not. Each of the 8 pages gets its own researched content, angle, and keyword focus so pages don't cannibalise each other in search.
3. **AI and traditional SEO are the same discipline now.** Every change (schema, content depth, FAQs, reviews) helps Google's classic ranking, Google AI Overviews, and chat assistants simultaneously. We are not building two separate strategies.
4. **Real data only.** Reviews, ratings, and testimonials must come from genuine sources (Google Business Profile, MyJobQuote) — no invented reviews or numbers.
5. **British English, UK trade terminology, B2B/consumer tone** throughout all new copy, matching the existing site voice.

**What was done, and why:**

1. **Verified the real starting point before writing anything**, rather than trusting the original audit's word for it a second time. Read every current `page.tsx` and `layout.tsx` metadata block, then ran the dev server and `curl`'d the actual rendered `<title>` tag on every route — which surfaced a genuine bug worth fixing regardless of copy: the homepage's title was rendering as exactly `"Fast, Professional Drainage Services"` with **no** `" | Jet Now Drainage"` suffix, while every other page correctly got the suffix from the root layout's `title.template`. Traced this to Next.js's own title-resolution source (`node_modules/next/dist/lib/metadata/resolvers/resolve-title.js`, checked directly per this repo's instruction to verify against the installed version rather than assume): a parent layout's `title.template` only applies to descendant segments, and the homepage's `page.tsx` sits at the *same* segment as the root layout that defines the template — so it silently never applied. This exactly matches what Jack's live Semrush check independently found (36 characters, no brand) — good confirmation the repo and the live site are the same code.
2. **Researched current (2026) best practice before writing a word of copy**, rather than working from older training-data assumptions: title tags (50–60 characters / ~580–600px, pixel width not character count is the real constraint, front-load the primary keyword, brand at the end); meta descriptions (140–160 characters, front-load the first ~120 for mobile truncation, include a verifiable fact/number and a named entity for AI-citation extractability — the emerging "GEO" criteria alongside classic CTR-focused SEO); and local-trade title formulas specifically (`[Primary service] + [location] + [brand]`, avoiding stuffing every coverage area into one title).
3. **Built `src/lib/metadata.ts`** — a `buildPageMetadata()` helper, following the same "single source of truth" pattern as `site-config.ts`/`JsonLd.tsx` from earlier phases. It uses `title: { absolute: ... }` rather than a plain string specifically to sidestep the template bug found in step 1: `absolute` bypasses the template entirely (confirmed in the same Next.js source file), giving every page a fully hand-tuned, predictable title rather than a mechanical prefix/suffix. It also builds that page's `openGraph` and `twitter` blocks from the same title/description.

### Phase 9 — Performance & Core Web Vitals (new, ~1 day — not part of Dan's original audit)

Surfaced on 05/07/2026 from a Semrush SEO Checker run and a Google PageSpeed Insights run Jack did against the live site (`docs/SEOResults/presemrush.txt`, `docs/SEOResults/preDevInsights.txt`), while gathering data for Phase 3. This is a real, severe, and entirely separate issue from anything Dan's report raised — flagged here for a decision, **not implemented yet**, since it wasn't this session's scope.

1. **The core problem — one oversized, unoptimised image.** The homepage's hero background, used as the `<video poster>` (`JN-HomeHero.jpg`), is a 6.3MB JPEG at 1724×2667px, not compressed and not served as a modern format (WebP/AVIF) or at a responsive size. 2. **Why this belongs in an SEO plan, not just a UX nice-to-have:** Core Web Vitals (LCP specifically) are a confirmed Google ranking factor, and slow-loading pages are also simply harder for crawlers — including AI crawlers — to fetch and process within their time budgets. Everything in Phases 1–8 improves what search engines and AI tools *understand* about the site; this affects whether they can load it quickly enough to want to.

The one intentionally empty `alt=""` (the floating WhatsApp button's icon) is correct practice, not a gap — the parent link already carries a proper `aria-label`, so the icon itself is correctly marked decorative rather than double-announced to screen readers. Semrush's original finding almost certainly predates Phase 4/5, when the whole services section was one page with far fewer images. Nothing left to do here.

#### ✅ Phase 9 — mostly complete (05/07/2026)

**1. Hero LCP fix — done.** The homepage `<video poster>` was pointed straight at the raw 6.2MB `JN-HomeHero.jpg`. A `<video poster>` is a plain HTML attribute, not a Next.js `<Image>`, so — unlike every other image on the site — it was never passing through Next's image optimisation pipeline at all. Fixed by pointing it at Next's own `/_next/image` endpoint instead (`?url=...&w=1920&q=75`), the same endpoint `next/image` calls internally. Verified directly against the running dev server: **6.2MB down to 188KB (JPEG fallback) or 127KB (WebP, for supporting browsers) — a 97–98% reduction**, with no visible quality loss at the rendered size. This was the dominant cause of the site's very poor LCP score and is now closed.

**2. Sitewide button contrast — done.** Confirmed by calculation (and independently flagged by a live Lighthouse accessibility audit) that white button text on the site's two primary accent fills failed WCAG AA's 4.5:1 normal-text threshold: `#ea591b` orange measured 3.53:1, `emerald-500` green measured 2.54:1. Neither button's text is large enough (18.66px bold or 24px regular) to qualify for the relaxed 3:1 "large text" threshold, so 4.5:1 is the correct bar. Fixed by darkening both to `#b14110` (5.78:1) and `emerald-700` (5.49:1) — same hues, same visual language, now passing — applied consistently across every hero and CTA button on the site (13 files: every page hero, `HomeCta`, `AreaPageCta`, `ServicePageCta`, and the contact form's submit button, including its hover-shadow tint colour). While checking this, also found and fixed the same underlying failure on three components using the *other* brand colour (`--brand-primary`, `#408ab9` blue) as a solid white-text fill rather than as text/border colour: `HomeStatsStrip`, the mobile nav's "Call" button, and the service-card flip overlay on `/services` — all measured 3.78:1, all fixed the same way with a darkened `#235f82`, applied locally rather than by changing the shared `--brand-primary` variable (which is used elsewhere, safely, as plain text/border colour and shouldn't be recoloured sitewide for a fix only needed in these solid-fill cases).

**3. Logo aspect ratio — done, but turned out to be a non-visual fix.** Checking this properly (rendering the live page and reading the actual computed box size, not just the HTML attributes) showed the logo was **not** actually stretched on screen: `DesktopNavbar`/`MobileNavbar`/`SiteFooter`/`ComingSoonScreen` all render the `<Image>` with `className="h-auto w-auto"`, which overrides the (wrong) `width`/`height` props for actual layout and lets the browser display the image at its own correct proportions. The `width`/`height` props were still wrong relative to the real file (972×350) — corrected to 183×66 / 161×58 / 161×58 / 267×96 respectively — which matters for accurately reserving layout space *before* the image loads (a minor layout-shift/Core Web Vitals contributor) and clears the Lighthouse "aspect ratio" warning, but this was never a visible-distortion bug in practice.

**4. Background video — done (05/07/2026).** No video tooling (`ffmpeg` or similar) is available in this environment, so Jack re-encoded `Trimmed3.mp4` himself using FreeConvert's online compressor (H.264, CRF 27, slowest compression speed — recommended here for a muted, looping, permanently-overlaid background clip where fine detail is never actually visible). Result: **4.1MB → 1.4MB, a 65% reduction**, swapped in as `Trimmed3Compressed.mp4` and wired into `HomeHero.tsx`; the original `Trimmed3.mp4` was deleted from `public/` since nothing referenced it anymore. Combined with the poster-image fix above, both pieces of the homepage's hero media are now properly optimised.

Nothing left open in this phase — the X/LinkedIn `sameAs` question raised in the original findings above has moved to the handover document (Section 8) rather than sitting here as a plan blocker.
